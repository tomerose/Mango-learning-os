```tsx

  <AlertCircle className="h-4 w-4" />
```

Paired with a **retry button** that triggers refetch/reset — never just the banner alone.

**D. Document Card Pattern — Compound Card**
```tsx

    <CardTitle>document-name.pdf</CardTitle>

  </CardContent>
  <CardFooter>

  </CardFooter>

```

**E. Four-State Conditional Render Pattern** (industry consensus 2025-2026)
```
isLoading  → <SkeletonGrid />
isError    → <Alert variant="destructive" /> + RetryButton
isEmpty    → <Empty /> + CTA
hasData    → <Content />
```
No `else` chain — explicit early returns per state.

- Skeleton (already installed), Empty compound (add via CLI), Card (already installed).
- **Action**: Create `src/components/shared/states.tsx` barrel exporting `LoadingSkeleton`, `ErrorState`, `EmptyState` as thin wrappers around shadcn primitives. Keep `<Empty>` as v4's compound component — cleaner than DIY.
- Document card: already fits MangoOS document output use case. Use Card compound + add metadata row.

## 2. Vercel AI SDK — Streaming & Generation States

### Reference 2.1: Vercel AI SDK Official (vercel/ai)

- **Repo**: https://github.com/vercel/ai  
- **Docs**: https://ai-sdk.dev/docs  
- **Stars**: 15k+ | **License**: Apache 2.0

#### Key Patterns Extracted

**A. useChat — Full Lifecycle with Status Enum**

The `status` field is the single source of truth for UI state. Five statuses: `"ready"` | `"submitted"` | `"streaming"` | `"error"` | `"idle"`.

```tsx
const { messages, sendMessage, status, stop, error, regenerate } = useChat({
  transport: new DefaultChatTransport({ api: '/api/chat' }),
});

// Loading: status === 'submitted' or status === 'streaming'
{(status === 'submitted' || status === 'streaming') && (
  <div>
    {status === 'submitted' && <Spinner />}
    <button onClick={() => stop()}>Stop</button>
  </div>
)}

// Error: error object + regenerate()
{error && (
  <>
    <div className="text-red-500">An error occurred.</div>
    <button onClick={() => regenerate()}>Retry</button>
  </>
)}

// Input disabled during streaming
<input disabled={status !== 'ready'} />
<button type="submit" disabled={status !== 'ready'}>Submit</button>
```

**B. Smart Loading — Only Show Skeleton When No Parts Yet**

Don't show "Loading..." when status is streaming but content has already started arriving:

```tsx
const lastMessage = messages.at(-1);
const showLoader = status === 'streaming'
  && lastMessage?.role === 'assistant'
  && lastMessage?.parts?.length === 0;
```

This prevents flickering between content and a loading indicator.

**C. Stream Resumption (useChat `resume: true`)**

For network interruptions — the stream resumes automatically without losing generated content:

```tsx
const { messages, sendMessage, status } = useChat({
  id: chatId,
  resume: true,
  transport: new DefaultChatTransport({
    prepareSendMessagesRequest: ({ id, messages }) => ({
      body: { id, message: messages[messages.length - 1] },
    }),
  }),
});
```

**D. Message Parts — Multi-Modal Rendering**

Messages are structured as `parts[]`, not raw strings. Each part has a `type`:

```tsx
{messages.map(m => (
  <div key={m.id}>
    <div className="font-bold">{m.role}</div>
    {m.parts.map((part, i) => {
      if (part.type === "text") return <span key={i}>{part.text}</span>;
      if (part.type === "tool-invocation") return <ToolCard key={i} {...part} />;
      if (part.type === "file") return <FilePreview key={i} {...part} />;
      return null;
    })}
  </div>
))}
```

#### MangoOS Assessment: **ADAPT**

- MangoOS has AI features — document Q&A, summarization.
- `useChat` with status-driven UI is the canonical pattern. No need to reinvent.
- **Action**: 
  - Use `status === 'streaming'` + lastMessage.parts.length check for smart loading (Pattern B).
  - Keep the `stop() / regenerate()` button pair always visible during active generation.
  - Enable `resume: true` for long document generations.
  - Map message `parts[]` to MangoOS-specific UI: text parts render inline, tool-invocation parts become structured cards.

## 3. TanStack Query — Mutation Lifecycle

### Reference 3.1: TanStack Query Official (TanStack/query)

- **Repo**: https://github.com/TanStack/query  
- **Docs**: https://tanstack.com/query/latest  
- **Stars**: 43k+ | **License**: MIT

#### Key Patterns Extracted

**A. Mutation Lifecycle — onMutate → onError → onSettled**

The gold-standard pattern for save/mutate operations:

```tsx
const useUpdateDocument = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: (doc) => fetch(`/api/documents/${doc.id}`, {
      method: 'PATCH',
      body: JSON.stringify(doc),
    }).then(r => { if (!r.ok) throw new Error('Failed'); return r.json(); }),

    // 1. BEFORE mutation: snapshot + optimistic update
    onMutate: async (newDoc) => {
      await queryClient.cancelQueries({ queryKey: ['documents'] });
      const previous = queryClient.getQueryData(['documents']);
      queryClient.setQueryData(['documents'], (old) =>
        old.map((d) => (d.id === newDoc.id ? newDoc : d))
      );
      return { previous }; // context for rollback
    },

    // 2. ON ERROR: rollback to snapshot
    onError: (err, newDoc, context) => {
      queryClient.setQueryData(['documents'], context?.previous);
      toast.error('Save failed. Reverted.');
    },

    // 3. ALWAYS (success or error): refetch to sync
    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ['documents'] });
    },
  });
};

// In component:
const { mutate, isPending, isError } = useUpdateDocument();
<Button disabled={isPending} onClick={() => mutate(doc)}>
  {isPending ? <Spinner /> : null} Save
</Button>
```

**B. Retry Configuration — Query-Level Defaults**

```tsx
const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      retry: 3,
      retryDelay: (attempt) => Math.min(1000 * 2 ** attempt, 30000), // exponential backoff, max 30s
    },
    mutations: {
      retry: 0, // NEVER auto-retry non-idempotent mutations
    },
  },
});
```

**C. Reusable Query Options (v5 pattern)**

```tsx
// query-options.ts
export const documentsQueryOptions = {
  queryKey: ['documents'],
  queryFn: fetchDocuments,
  staleTime: 5 * 60 * 1000,
};
// In component: useQuery(documentsQueryOptions)
// For prefetch: queryClient.prefetchQuery(documentsQueryOptions)
```

#### MangoOS Assessment: **ADAPT**

- MangoOS document CRUD needs exactly this: save with rollback, retry on network failure.
- **Action**:
  - Wrap all document mutations (create, update, delete, rename) in the onMutate→onError→onSettled lifecycle.
  - Set `retry: 3` for queries, `retry: 0` for mutations (non-idempotent POSTs must not double-create).
  - Extract reusable `queryOptions` for shared query keys.
  - Derive UI loading states from `isPending`, not from separate `useState` flags.

### Reference 3.2: TanStack Query v5 — useSuspenseQuery + ErrorBoundary

- **Source**: Official docs, confirmed by community practice

#### Key Pattern

```tsx
// v5: useSuspenseQuery guarantees data is never undefined
const { data } = useSuspenseQuery({
  queryKey: ['document', id],
  queryFn: () => fetchDocument(id),
  retry: 3,

```

This eliminates conditional `isLoading` / `isError` branches in every component — React boundaries handle both.

- **Action**: Use `useSuspenseQuery` for main document load. Keep `useQuery` (non-suspense) for secondary data where you want inline error handling.

## 4. Zustand Persist — Local State Persistence

### Reference 4.1: Zustand Official (pmndrs/zustand)

- **Repo**: https://github.com/pmndrs/zustand  
- **Docs**: https://zustand.docs.pmnd.rs/reference/integrations/persisting-store-data  
- **Stars**: 50k+ | **License**: MIT

#### Key Patterns Extracted

**A. Guest User / Draft Persistence**

```tsx
import { create } from 'zustand';
import { persist, createJSONStorage } from 'zustand/middleware';

interface GuestState {
  guestId: string;
  draftContent: string | null;
  intent: 'browsing' | 'uploading' | 'analyzing';
  setDraft: (content: string) => void;
  clearDraft: () => void;
  upgradeToUser: (userId: string) => void;
}

export const useGuestStore = create<GuestState>()(
  persist(
    (set) => ({
      guestId: crypto.randomUUID(), // generated ONCE, then persisted
      draftContent: null,
      intent: 'browsing',

      setDraft: (content) => set({ draftContent: content, intent: 'analyzing' }),
      clearDraft: () => set({ draftContent: null, intent: 'browsing' }),
      upgradeToUser: (userId) => {
        // Merge guest data into user account, then clear
        set({ draftContent: null, intent: 'browsing' });
        useGuestStore.persist.clearStorage();
      },
    }),
    {
      name: 'mangoos-guest-storage',
      storage: createJSONStorage(() => localStorage),
      partialize: (state) => ({
        guestId: state.guestId,
        draftContent: state.draftContent,
        intent: state.intent,
      }),
    }
  )
);
```

**B. Schema Migration (version + migrate)**

```tsx
persist(
  (set) => ({ newField: 0 }),
  {
    name: 'mangoos-preferences',
    version: 2, // bump when schema changes
    migrate: (persisted, version) => {
      if (version === 0) {
        // rename old field
        persisted.newField = persisted.oldField;
        delete persisted.oldField;
      }
      if (version === 1) {
        // add new field with default
        persisted.newFeature = false;
      }
      return persisted as MyState;
    },
  }
)
```

**C. SSR Hydration Control**

```tsx
// For Next.js App Router — avoid hydration mismatch
persist(
  (set) => ({ ... }),
  {
    name: 'mangoos-storage',
    skipHydration: true, // defer hydration to client
  }
)

// In _app or layout:
useEffect(() => {
  useStore.persist.rehydrate();
}, []);
```

**D. Cross-Tab Sync Warning**

`localStorage` is shared across tabs but Zustand does NOT auto-sync. Solutions:
- `window.addEventListener('storage', callback)` for cross-tab notification
- BroadcastChannel API for real-time sync

#### MangoOS Assessment: **ADAPT**

- MangoOS needs guest-mode draft persistence (documents being edited before login), user preferences (theme, layout), and recently-viewed items.
- **Action**:
  - Guest document drafts: persist with `partialize` (only `draftContent`, `guestId`).
  - Schema migration: start with `version: 1` to enable future evolution.
  - SSR: use `skipHydration: true` in all persisted stores.
  - Don't use for sensitive data (API keys, tokens) — localStorage is not encrypted.

### Reference 4.2: StackOverflow — Guest UUID + Persist Gotcha (July 2025)

- **URL**: https://stackoverflow.com/questions/79716227  
- **Issue**: Community-validated pattern for guest ID persistence

#### Key Finding

**Bug**: Without `set` parameter in the state creator, `crypto.randomUUID()` regenerates on every hydration cycle:

```tsx
// BROKEN — regenerates UUID on every refresh
persist((): UserState => ({ userId: crypto.randomUUID() }), { name: '...' })

// FIXED — UUID generated once, then persisted
persist((set) => ({ userId: crypto.randomUUID(), setUserId: (id) => set({ userId: id }) }), { name: '...' })
```

#### MangoOS Assessment: **ADAPT** — already reflected in Pattern 4.1A above.

## 5. File Upload — Best Practices

### Reference 5.1: react-simple-file-input (MIT, v2.1.0)

- **Repo**: https://github.com/greena13/react-simple-file-input  
- **Stars**: Small library, but pattern is production-grade

#### Key Patterns Extracted

**A. Declarative Validation Model**

Three validation hooks — check BEFORE reading, DURING read, and AFTER read:

| Prop | Stage | Purpose |
|------|-------|---------|
| `cancelIf={validator}` | Pre-read | Validate File object (type, size) before `readAsDataURL`. Returns truthy to cancel. |
| `abortIf={condition}` | Mid-read | Check on each progress tick. Returns truthy to call `reader.abort()`. |
| `onError={handler}` | Read failure | Catches native `FileReader` errors. |

```tsx
const ALLOWED_TYPES = ["application/pdf", "image/png", "image/jpeg"];
const MAX_SIZE = 10 * 1024 * 1024; // 10MB

function fileIsInvalid(file: File) {
  if (!ALLOWED_TYPES.includes(file.type)) return "Unsupported file type.";
  if (file.size > MAX_SIZE) return "File too large (max 10MB).";
  if (file.size === 0) return "File is empty.";
  return null; // null = valid
}

<FileInput
  readAs="dataUrl"
  cancelIf={fileIsInvalid}
  onCancel={showErrorToast}     // fires when cancelIf returns truthy
  onError={handleReaderError}   // fires on native FileReader error
  onLoad={(e, file) => uploadBase64(e.target.result)}
/>
```

**B. FileReader Lifecycle Wrapper (Promise Pattern)**

Encapsulate the entire FileReader lifecycle in a Promise for clean async/await:

```tsx
function readFileAsDataURL(file: File): Promise<string> {
  return new Promise((resolve, reject) => {
    const reader = new FileReader();
    reader.onload = () => {
      if (reader.result && typeof reader.result === 'string') {
        resolve(reader.result.split(',')[1]); // strip MIME prefix
      } else {
        reject(new Error('Failed to read file.'));
      }
    };
    reader.onabort = () => reject(new Error('Read aborted.'));
    reader.onerror = () => reject(new Error('File read error.'));
    reader.readAsDataURL(file);
  });
}
```

**Validation Check Sequence** (order matters):

1. Empty file → `file.size === 0` → reject BEFORE reading
2. Unsupported type → `!ALLOWED_TYPES.includes(file.type)` → reject BEFORE reading
3. Too large → `file.size > MAX_SIZE` → reject BEFORE reading
4. `reader.readAsDataURL(file)` → only after all checks pass

#### MangoOS Assessment: **ADAPT**

- MangoOS handles document uploads (PDF, images). Needs type + size validation.
- **Action**:
  - Adopt the Promise-wrapped FileReader pattern (5.1B).
  - Adopt the pre-read validation sequence (5.1B step 1-3).
  - Use `onabort`, `onerror`, and `onload` — all four must be bound (never leave `onerror` unhandled).
  - Strip MIME prefix after read: `reader.result.split(',')[1]`.
  - Don't use react-simple-file-input library itself — extract the pattern into MangoOS's own `useFileReader` hook.

### Reference 5.2: KendoReact Upload — FileReader in Production

- **Source**: StackBlitz fork of KendoReact Upload component  
- **URL**: https://stackblitz.com/edit/react-u2kbu9-ywthns

#### Key Pattern

The same Promise-wrapped FileReader, but additionally:
- Checks `currentFile.validationErrors` before attempting read
- Uses `reader.onprogress` with `event.loaded / event.total` for percentage-based progress
- Handles all four termination paths: success, abort, error, and invalid-result

```tsx
reader.onprogress = (e) => {
  if (e.lengthComputable) {
    const percent = Math.round((e.loaded / e.total) * 100);
    setProgress(percent);
  }
};
```

#### MangoOS Assessment: **ADAPT-WITH-MOD**

- Progress tracking is useful for large document uploads.
- **Action**: Add `onprogress` handler when file > 1MB. Skip progress UI for small files (< 1MB) to reduce noise.

## 6. Next.js App Router — Dashboard + Document Output Patterns

### Reference 6.1: next-shadcn-admin-dashboard (arhamkhnz)

- **Repo**: https://github.com/arhamkhnz/next-shadcn-admin-dashboard  
- **Stars**: 2,450 | **License**: MIT  
- **Stack**: Next.js 16 (App Router), shadcn/ui, Tailwind CSS v4, Zustand, TanStack Table, Recharts, Zod + React Hook Form

#### Key Patterns Extracted

**A. Feature-Based Directory Structure**

```
src/
  app/           # Next.js route definitions only (thin)
    dashboard/
      overview/
      products/
      users/
      kanban/
      chat/
  components/
    ui/          # shadcn primitives
    layout/      # Header, Sidebar, Shell
    themes/      # Theme toggle/config
  features/      # Feature modules (thick — all logic here)
    overview/    # Charts, metric cards
    products/    # Tables, forms (React Query)
    kanban/      # Drag-drop board (zustand + dnd-kit)
    chat/        # Conversations, bubbles, composer
  lib/           # Query client, utilities
  hooks/         # Shared hooks
  config/        # Nav config, data-table config, infobar
```

Key principle: `app/` is thin (route definitions), `features/` is thick (UI logic, state, queries).

**B. Zustand + Feature Co-location**

Stores live INSIDE their feature directory, not in a global `stores/` folder:
```
features/kanban/  → uses zustand for board/column/task state
features/notifications/ → uses zustand for notification center
features/chat/ → uses zustand for conversations
```

No monolithic global store. Each feature owns its state.

**C. Config-Driven UI**

Navigation, infobar, data-table columns are all driven by config objects in `src/config/`, not hardcoded in components. Enables RBAC: different roles get different sidebar items.

**D. Theme System**

Multiple theme presets (Tangerine, Neo Brutalism, Soft Pop, etc.) implemented as CSS variable swaps. Dark/light mode via `next-themes`. No hardcoded colors.

#### MangoOS Assessment: **ADAPT**

- MangoOS is a dashboard + document output app. This project's structure is the canonical reference.
- **Action**:
  - Adopt feature-based directory structure: `features/documents/`, `features/chat/`, `features/analysis/`.
  - Keep `app/` thin. Route definitions only.
  - Co-locate stores with features. No global store dump.
  - Use config-driven navigation (sidebar items from `config/nav.ts`).
  - Adopt the multiple-theme system for MangoOS branding flexibility.
