# MangoOS 商业计划书 — 数据缺口清单

**创建日期：** 2026年6月9日
**用途：** 列出商业计划书中缺失的真实数据，指导下一步数据收集工作。

---

## 一、用户数据（全部缺失）

| 数据项 | 当前状态 | 需如何获取 |
|--------|:--:|------|
| 总注册用户数 | ❌ 无数据 | 在 Supabase `auth.users` 表中统计，或在应用内接入统计 |
| 月活跃用户数（MAU） | ❌ 无数据 | 需接入统计工具（如 Vercel Analytics、Plausible、Umami）或自建活跃记录表 |
| 日活跃用户数（DAU） | ❌ 无数据 | 同上 |
| 用户留存率（Day 1 / Day 7 / Day 30） | ❌ 无数据 | 需埋点统计用户回访行为 |
| 游客 vs 注册用户比例 | ❌ 无数据 | 从 Supabase 和 cookie 记录可推算 |
| 用户来源渠道分布 | ❌ 无数据 | 需 UTM 参数或渠道统计 |
| Pro / 免费用户比例 | ❌ 无数据 | 从 `mango_codes` 表可部分推算 |

**优先级：** 🔴 高。用户数据是投资人最关心的基础指标。

---

## 二、使用行为数据（全部缺失）

| 数据项 | 当前状态 | 需如何获取 |
|--------|:--:|------|
| 平均单次会话时长 | ❌ 无数据 | 前端埋点 session duration |
| 最常用的功能模块排名 | ❌ 无数据 | 页面访问量统计 |
| Agent 任务生成次数 | ❌ 无数据 | `agent_runs` 表可统计（如果迁移已执行） |
| Exam Review 生成次数 | ❌ 无数据 | 需在 API 端增加计数器 |
| PDF/文档导出次数 | ❌ 无数据 | `outcome_exports` 表可统计 |
| 用户任务输入的常见类型 | ❌ 无数据 | 分析 `agent_runs.prompt` 字段 |
| 移动端 vs 桌面端比例 | ❌ 无数据 | 前端 User-Agent 统计 |

**优先级：** 🔴 高。使用行为数据验证产品是否真的在"被使用"。

---

## 三、质量数据（部分有数据）

| 数据项 | 当前状态 | 需如何获取 |
|--------|:--:|------|
| Agent 生成平均质量分数 | ⚠️ 部分 | `agent_runs.quality_score` 有结构但需积累数据 |
| 质量不合格率（<90 分比例） | ⚠️ 部分 | 同上 |
| 自动加深成功率（第 2 轮达标率） | ❌ 无数据 | 需在 Agent 执行逻辑中记录 |
| 用户对生成内容的满意度 | ❌ 无数据 | 需增加"有用/无用"或 1-5 星评分 UI |
| Admin 审核通过/驳回率 | ❌ 无数据 | Admin Review 页面有设计但无累积统计 |

**优先级：** 🟡 中。质量数据是核心竞争壁垒的证明。

---

## 四、付费与收入数据（全部缺失）

| 数据项 | 当前状态 | 需如何获取 |
|--------|:--:|------|
| 付费转化率 | ❌ 无数据 | 需上线真实付费实验 |
| 用户付费意愿（问卷/访谈） | ❌ 无数据 | 可在种子用户中做快速调研 |
| 芒果码激活量 | ⚠️ 部分 | `mango_codes` 表可查询 `used_at` 字段 |
| Pro 用户续费率 | ❌ 无数据 | 需真实付费系统上线后统计 |
| 客单价接受度 | ❌ 无数据 | 可做价格敏感度测试（问卷或 A/B） |
| 不同场景付费意愿差异 | ❌ 无数据 | 期末场景 vs 日常场景可分别测试 |

**优先级：** 🔴 高。付费验证是商业计划书从"假设"到"数据驱动"的关键一步。

---

## 五、成本数据（部分有数据）

| 数据项 | 当前状态 | 需如何获取 |
|--------|:--:|------|
| DeepSeek API 月调用量 | ❌ 无数据 | 查看 DeepSeek Dashboard 或做 API 日志统计 |
| 单次 Agent 生成的 token 成本 | ❌ 无数据 | 在 Agent API 中记录每次调用的 token 数 |
| 单次 Exam Review 的完整成本 | ❌ 无数据 | 检索 + AI 生成的综合成本 |
| 免费用户人均成本 | ❌ 无数据 | 总成本 / 活跃免费用户数 |
| Vercel / Supabase 当前用量 | ⚠️ 部分 | Vercel Dashboard 和 Supabase Dashboard 可查 |
| 单用户服务成本（含基础设施） | ❌ 无数据 | 总月成本 / MAU |

**优先级：** 🟡 中。单位经济学（Unit Economics）是投资人必问数据。

|--------|:--:|------|

| 用户从竞品迁移的意愿 | ❌ 无数据 | 可做"如果用 MangoOS 替代 XX 工具，你最在意什么" |

**优先级：** 🟢 低。竞品数据可从公开渠道逐步收集，不是当前最紧迫的缺口。

## 八、案例与见证（缺失）

| 数据项 | 当前状态 | 需如何获取 |
|--------|:--:|------|
| 真实用户案例（课程+生成结果+反馈） | ❌ 无数据 | 校园内测期间重点收集 |
| 用户推荐语 / 使用评价 | ❌ 无数据 | 内测期间收集 |
| 生成内容范例展示 | ⚠️ 部分 | 可自测生成几个真实课程的讲义作为展示 |

**优先级：** 🔴 高。案例和数据故事是打动投资人的最有力材料。

## 十、行动计划：优先补哪些数据

### 立即（本周）
1. **接入 1 个轻量统计工具**（如 Umami 自部署或 Plausible），开始收集页面访问和基本行为数据
2. **自测生成 3-5 个真实课程的期末复习讲义**，保存为展示案例
3. **在 Agent API 中加 token 计数**，记录每次调用的实际成本
4. **统计 Supabase 现有数据**：用户数、agent_runs 记录数、生成次数

### 短期（2-4 周）
5. **启动校园小范围内测**（第三自习室 + 同学），收集 20+ 使用案例
6. **做一次快速用户调研**（5-10 人，15 分钟访谈），了解付费意愿和使用场景
7. **整理竞品定价和融资数据表**，完善竞争格局分析

### 中期（1-3 个月）
8. **上线真实付费实验**（微信/支付宝），获取付费转化数据
9. **收集用户推荐语和使用案例**
10. **建立定期数据复盘机制**（月度关键指标邮件）

## 附：数据缺口严重程度总览

| 类别 | 严重程度 | 说明 |
|------|:--:|------|
| 用户数据 | 🔴 **严重** | 无任何真实用户数据，商业计划书完全依赖假设 |
| 使用行为 | 🔴 **严重** | 无法证明产品被真实使用 |
| 付费验证 | 🔴 **严重** | 商业模式未经任何验证 |
| 用户案例 | 🔴 **严重** | 没有能展示给投资人的真实案例 |
| 成本数据 | 🟡 中等 | API 成本可快速补充，基础设施成本已可查 |
| 质量数据 | 🟡 中等 | 数据结构已就绪，需积累使用量 |
| 竞品数据 | 🟢 较轻 | 可从公开渠道逐步补充 |
